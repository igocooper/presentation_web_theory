TCP

TCP - Transmission Control Protocol

  • Встановлює TCP з'єднання шляхом потрійного рукостискання
  • Конторлює передачу пакетів данних.
  • Контролює навантаження на мережу

Потрійне рукостискання:

Послідовний обмін пакетами між клієнтом і сервером, що дозволяє встановити TCP зв'язок для подальшої передачі данних

  • SYN — Клієнт генерує випадкове число, наприклад, x, і відправляє його на сервер.
  • SYN ACK — Сервер підтверджує цей запит, відправивши назад пакет ACK, з випадковим числом, і число x + 1 отримане від клієнта.
  • ACK — клієнт збільшує число y і посилає y + 1 на сервер.

UDP

UDP - User Datagram Protocol

UDP — транспортний протокол для передачі повідомленнь-датаграм без необхідності встановлення з'єдняння в IP-мережі.

TCP vs UDP

  • TCP гарантує доставку пакетів даних в незмінному вигляді, в правильній послідовності і без втрат, UDP нічого не гарантує.
  • TCP вимагає заздалегідь встановленого з'єднання, UDP з'єднання не вимагає.
  • UDP забезпечує більш високу швидкість передачі даних.
  • TCP надійніший і здійснює контроль над процесом обміну даними.
  • UDP краще для програм, що відтворюють потокове відео, відеофонії і телефонії, мережевих ігор.

HTTP - Hyper Text Transfer Protocol

HTTP - це протокол прикладного рівня, реалізований поверх протоколу TCP / IP.

HTTP versions:

  • HTTP / 0.9 - 1991
  • HTTP/1.0 — 1996
  • HTTP/1.1 – 1999
  • HTTP/2 — 2015

HTTP / 0.9 перший стандарт

Клієнт міг зробити лише GET запит зразка:

GET /index.html

Server response:

			(response body)
			(connection closed)

Ось так виглядав простий запит по протоколу HTTP / 1.0:

			GET / HTTP/1.0
			Host: kamranahmed.info
			User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10_10_5)
			Accept: */*

Ось так виглядала проста відповідь сервера по протоколу HTTP / 1.0:

			HTTP/1.0 200 OK
			Content-Type: text/plain
			Content-Length: 137582
			Expires: Thu, 05 Dec 1997 16:00:00 GMT
			Last-Modified: Wed, 5 August 1996 15:55:28 GMT
			Server: Apache 0.84

			(response body)
			(connection closed)

HTTP / 1.0 мінуси

  • Неможливо надіслати декілька запитів під час одного з'єднання.
  • Кожен додатковий ресурс = нове TCP з'єднання
  • TCP з'єднання вимагає 3-ного рукостискання після якого настає повільний старт
  • Не передбачена підтримка станів.
  • Cервер не зберігає інформації про клієнта, тому кожному запиту доводиться включати в себе всю необхідну серверу інформацію
  • Клієнт посилає дані, що повторюються, зайве перевантажуючи мережу.

Connection: keep-alive

"Эй, дружище, не закрывай это соединение, оно нам ещё пригодится"

HTTP / 1.1

  • Нові HTTP-методи: PUT, PATCH, HEAD, OPTIONS, DELETE.
  • Ідентифікація хостів: У HTTP / 1.0 заголовок Host не був обов'язковим, а HTTP / 1.1 зробив його таким.
  • Постійні з'єднання: Для закриття з'єднання необхідно надіслати заголовок Connection: close
  • Потокова передача данних: Задопомогою заголовку Content-Length:
  • Кешування
  • Клієнтські Cookies

HTTP / 1.1 недоліки

  • HTTP / 1.1 підтримує лише одне зовнішнє з'єднання в будь-який момент часу
  • head-of-line blocking
  • Неможливо використовувати потокову передачу данних, якщо данні динамічні.

Streaming Dynamic data issue

Chunked encoding - механізм розбиття інформації на невеликі частини (chunks) і їх передачу.

SPDY

SPDY — 2009

  • Мультиплексування
  • Зжимання данних
  • Пріоритизація
  • Безпека

У 2015 в Google вирішили, що не повинно бути двох конкуруючих стандартів, і об'єднали SPDY з HTTP, давши початок HTTP / 2.

HTTP / 2

HTTP / 2

  • Бінарний замість текстового
  • Мультиплексування - передача декількох асинхронних HTTP-запитів по одному TCP-з'єднання
  • Стиснення заголовків методом HPACK
  • Server Push - кілька відповідей на один запит
  • Пріоритизація запитів
  • Безпека

Бінарні повідомлення:

Бінарні повідомлення швидше розбираються автоматично, але, на відміну від HTTP / 1.x, не зручні для читання людиною. Основні складові HTTP / 2 - фрейми (Frames) і потоки (Streams).

Зжимання заголовіків методом HPACK:

Якщо ми постійного звертаємося до сервера з одного і того ж клієнта, то в заголовках раз по раз посилається величезна кількість повторюваних даних.

Server Push

Сервер, знаючи, що клієнт збирається запросити певний ресурс, може відправити його, не чекаючи запиту. Сервер відправляє спеціальний фрейм PUSH_PROMISE

Пріоритизація

Клієнт може призначити пріоритет потоку, додавши інформацію про пріоритет у фрейм HEADERS, яким відкривається потік. Будь-який інший час клієнт може відправити фрейм PRIORITY, який змінює пріоритет потоку.

Безпека

HTTP/2 повинен бути реалізованный поверх TLS, хоч спецмфікація цього не вимагає.

HTTPS

  • конфіденційність
  • цілісність
  • достовірність

Конфіденційність

    Щоб дві сторони могли спілкуватися, вони повинні прийти до згоди з двох питань:
  • Який алгоритм (шифрувальну функцію) використовувати в комунікації.
  • Які параметри, паролі або правила (тобто секрет) будуть використовуватися для обраного методу.

Шифрування

  • симетричне - Обидві сторони володіють загальним секретним ключем.
  • асиметричне - Одна зі сторін володіє парою з публічного і секретного ключів, представляючи основу інфраструктури публічних ключів (public key infrastructure, PKI).

Цілісність

Для забезпечення цілісної передачі інформації використовуються алгоритми дайджест повідомлення. Обчислення кодів аутентифікації повідомлень (MAC) для кожного повідомлення при обміні.

    Tаким чином неможливо:
  • змінити повідомлення, не затронувши тег
  • згенерувати однаковий тег для двох різних повідомлень
  • перехопити процес і отримати оригінальне повідомлення з тега

Перевірка аутентичності

Виконується третьою стороною - центром сертифікації (CA)

CA випускає сертифікат з підтвердженням того, що доменне ім'я example.com (унікальний ідентифікатор) пов'язано з публічним ключем XXX. У деяких випадках CA також перевіряє, що конкретна компанія контролює цей домен.

Види сертифікатів

  • Підтверджений домен (DV)
  • Розширене підтвердження (EV)
  • Підтверджена організація (OV)
